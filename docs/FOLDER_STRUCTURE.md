# Folder Structure вЂ” SpeakUp Connect

---

## Overview

The folder structure follows **feature-based clean architecture**. Each feature is self-contained with its own data, domain, and presentation layers. Shared code lives in `core/` and `shared/`.

This structure is chosen because:
- Features can be developed, tested, and modified in isolation
- AI-assisted development is more predictable with consistent patterns
- Onboarding new contributors is faster with explicit folder conventions
- Long-term maintenance is easier when concerns are cleanly separated

## Folder Conventions

### `core/` вЂ” Infrastructure, Not Business Logic

The `core/` folder contains infrastructure that the entire app depends on but that has no business logic. This includes:
- **Theme system** вЂ” Colors, typography, ThemeData
- **Router** вЂ” All route definitions and guards
- **Constants** вЂ” App-wide string and numeric constants
- **Errors** вЂ” Custom exception and failure types
- **Extensions** вЂ” Dart extension methods
- **Utils** вЂ” Pure functions (validators, formatters)

**Rule:** `core/` code must not import from `features/`.

### `config/` вЂ” App Configuration

Environment configuration and Firebase initialization live here. This is intentionally separate from `core/` because config changes (e.g., switching Firebase projects) should not require touching infrastructure code.

### `features/` вЂ” Business Features

Each feature is a self-contained vertical slice of the application:
- A feature can depend on `core/`, `config/`, and `shared/`
- A feature **must not** import from another feature directly
- Inter-feature communication happens via shared providers or the router

### `shared/` вЂ” Cross-Feature UI & Models

Reusable widgets and models used by more than one feature live in `shared/`. The key rule is:
- If only one feature uses it, it belongs in that feature's `presentation/widgets/`
- If two or more features use it, it moves to `shared/widgets/`

## Architectural Decision Records

### ADR-001: Feature-Based Over Layer-Based

**Decision:** Organize code by feature, not by layer (no top-level `models/`, `widgets/`, `screens/` folders).

**Reason:** As the app grows, feature-based organization scales better. Finding all code related to "reports" means looking in one place (`features/reports/`), not scattered across multiple top-level directories.

### ADR-002: Clean Architecture Per Feature

**Decision:** Every feature has domain/data/presentation layers.

**Reason:** Decouples business logic from Firebase implementation details. If we ever migrate from Firebase to a different backend, only the `data/` layer needs to change.

### ADR-003: No Direct Feature-to-Feature Imports

**Decision:** Features cannot import each other directly.

**Reason:** Prevents tight coupling between features. If the reports feature needs auth state, it uses the Riverpod provider graph, not a direct import from `features/auth/`.

### ADR-004: Shared Widgets Must Be Generic

**Decision:** Widgets in `shared/widgets/` must be fully generic and not contain business logic.

**Reason:** Shared widgets should be reusable across any organization and any context without modification.
